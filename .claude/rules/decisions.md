| 15  | **The API boundary throws the `ApiError` envelope (a plain object), not an `Error` instance — and `@typescript-eslint/only-throw-error` is configured to `allow` exactly that one type (not disabled).** Do not "fix" this by making `ApiError` an `Error` subclass or by throwing `new Error()`.                                                                                                                                                                                             | The error contract with the Go API is the `{error, message}` envelope: the client narrows untrusted responses to it (`isApiError`), callers switch on `error.error`, and tests assert the envelope shape. | 16  | **Auth architecture: a Backend-for-Frontend (BFF) confidential OIDC client, not tokens in the browser.** A small Node/TypeScript service (`bff/`) performs Authorization Code + PKCE (S256), holds every access/refresh/ID token server-side, hands the browser only a `__Host-` session cookie, and reverse-proxies `/api/*` to go-api-template with `Authorization: Bearer <access token>` attached. Supersedes the "only the seam" framing of #1/#2.                                       | The BFF is the **top-ranked** architecture of the IESG-approved, RFC-Editor-queue BCP (§6.1) — cite it as Best Current Practice, not "just a draft". The browser never holds a token, so XSS cannot exfiltrate one. **CSP is unchanged: `connect-src 'self'` still holds** because the SPA only ever talks same-origin and the IdP round trip is a top-level navigation, not a fetch — the headline security win. The **Go API is unchanged and pattern-agnostic** (it still just validates a bearer): commit `dfb53b4`, untouched. | [draft-ietf-oauth-browser-based-apps §6.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-browser-based-apps), [RFC 9700 §2.1.1](https://www.rfc-editor.org/info/rfc9700), [RFC 7636](https://www.rfc-editor.org/info/rfc7636); `bff/src/`, `Caddyfile` |
