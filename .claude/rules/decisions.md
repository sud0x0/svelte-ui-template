# Settled decisions

Deliberate trade-offs already made in this template, each with a one-line
rationale and a source. **Do not relitigate these unless the user explicitly
asks.** A proposal that contradicts an entry here is a finding to flag against
this file — not a default to adopt. Record new trade-offs here.

| 1   | **OIDC intentionally not implemented — only the seam is.** Ship `VITE_AUTH_MODE`, the `auth.ts` stubs, `RouteGuard`, and the client 401 + CSRF seams; flipping to `bff` + filling stubs is the whole change.                                                                                                                                                                                                                                               | Auth requirements vary per deployment; ship the contract so the future BFF is a drop-in. Mirrors the Go template's auth-contract decision.                                                                                                      | [go-api-template decisions #13](https://github.com/sud0x0/go-api-template); `src/lib/api/auth.ts`, `src/lib/config.ts`                                                                                                              |
| 2   | **Intended model: BFF over browser-held tokens.** The Go API holds all tokens; the SPA holds none.                                                                                                                                                                                                                                                                                                                                                         | Strongest browser-app pattern. The browser-based OAuth-client (tokens in memory, PKCE in `sessionStorage`) is the recorded _weaker_ fallback for backend-less deployments.                                                                      | [draft-ietf-oauth-browser-based-apps](https://datatracker.ietf.org/doc/draft-ietf-oauth-browser-based-apps/), [RFC 9700](https://www.rfc-editor.org/info/rfc9700)                                                                   |
| 3   | **CSRF: double-submit `X-CSRF-Token` header** (over Fetch-Metadata), plus `__Host-` `SameSite=Strict` session cookie.                                                                                                                                                                                                                                                                                                                                      | Double-submit is framework-agnostic and trivial for the SPA to echo; `SameSite=Strict` alone is necessary-but-insufficient per OWASP. Swappable to `Sec-Fetch-Site` at the BFF without SPA changes.                                             | [OWASP CSRF](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html), [OWASP Session Management](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html) |
| 4   | **CSP: strict `script-src 'self'` via no-inline-script build; `style-src 'self' 'unsafe-inline'` compromise.** Achieved with `modulePreload.polyfill=false` + `assetsInlineLimit=0` — no CSP plugin.                                                                                                                                                                                                                                                       | A static SPA can't mint per-request nonces. Removing the only inline script lets `script-src` stay strict with no hashing. Vite injects code-split CSS via runtime `<style>`; relaxing _style_ (lower risk) beats relaxing _script_.            | [MDN CSP](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/CSP), [Vite build options](https://vite.dev/config/build-options)                                                                                                |

| 6   | **Routing: hand-rolled History-API router** (params, back/forward, 404, lazy `import()`) over a library.                                                                                                                                                                                                                                                                                                                                                   | Keeps runtime dependencies at zero; the router is ~120 lines and fully owned. A maintained runes-era router would be acceptable but adds a dependency to vet.                                                                                   | `src/lib/stores/router.svelte.ts`                                                                                                                                                                                                   |
| 7   | **State idiom: runes in `.svelte.ts`, plain accessors** — not `writable`. Store API only for external interop.                                                                                                                                                                                                                                                                                                                                             | One idiom across the codebase; avoids mixing legacy stores with runes. Persisted prefs go through a thin typed wrapper; auth state never persists.                                                                                              | `src/lib/stores/*.svelte.ts`, [Svelte 5 runes](https://svelte.dev/docs/svelte/what-are-runes)                                                                                                                                       |
| 8   | **Tests: Vitest 4 Browser Mode** (`vitest-browser-svelte` + the `@vitest/browser-playwright` provider, configured as `provider: playwright()`) + MSW (`msw/browser` Service Worker) for the API boundary; Playwright for E2E.
